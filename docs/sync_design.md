# 同期処理詳細設計書

本ドキュメントでは、Google カレンダーおよびフリカレの同期ロジックの具体的な実装仕様を定義しますわ。

## 1. データベース拡張

各ユーザーの同期状態を管理するため、`profiles` テーブルに以下のカラムを追加します。

- `google_refresh_token`: Google API のアクセストークンを更新するためのリフレッシュトークン。
- `last_sync_at`: 最終同期日時。

## 2. Google カレンダー同期フロー (Issue #1)

### 2.1 トークン取得と保存
- ユーザーがログインする際、`prompt=consent` および `access_type=offline` を指定して、Google からリフレッシュトークンを取得します。
- `auth/callback/route.ts` 内で、取得したリフレッシュトークンを `profiles.google_refresh_token` に保存します。

### 2.2 バックグラウンド同期
- `src/lib/sync/google.ts` にて、`google.auth.OAuth2` クライアントを初期化します。
- 保存されているリフレッシュトークンをセットし、必要に応じて `getAccessToken()` を呼び出して最新のトークンを取得します。
- `calendar.events.list` を呼び出し、直近 1 ヶ月分（または指定範囲）の予定を取得します。
- 取得したイベントを `CalendarEvent` 型にマッピングし、DB の `events` テーブルを更新します。

## 3. フリカレ同期フロー (Issue #2)

### 3.1 スクレイピング仕様
- 共有 URL（例: `https://fricare.com/share/...`）に対して `fetch` を実行します。
- Cheerio を用いて以下のセレクタを解析します（※実際の HTML 構造に基づき調整しますわ）：
  - `.calendar-day`: 各日付のセル
  - `.event-item`: 予定の要素
- 予定から「日付」「タイトル」「開始/終了時刻」を抽出します。
- 時刻の指定がない場合は「終日（is_all_day = true）」として扱います。

### 3.2 データ変換
- 抽出したデータを `CalendarEvent` 型に変換します。
- フリカレにはユニークな ID がない場合が多いため、`external_id` は「URL + 日付 + タイトル」のハッシュ値などで代用するか、ユーザー単位で全削除・再挿入を行います。

## 4. 共通オーケストレーション (orchestrator.ts)

- 全ユーザーをループし、有効な同期設定を持つソース（Google/Frica）を順次実行します。
- 同期の整合性を保つため、以下のステップで更新を行います：
  1. 特定ユーザーの既存イベント（該当ソース分）を削除。
  2. 新しいイベントを一括挿入（bulk insert）。
  3. `profiles.updated_at` を更新。

## 5. エラーハンドリング

- **Google トークン失効:** ログを記録し、`google_sync_enabled` を `false` に変更するか、ユーザーに再ログインを促すフラグを立てます。
- **スクレイピング失敗:** 構造変更の可能性があるため、詳細なエラーログを残し、管理者に通知できるようにします。

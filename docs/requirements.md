# Webアプリケーション要件定義書（第3版）

## 1. システム概要
特定のお二人が利用している「フリカレ」と「Googleカレンダー」の予定を、一つの画面上で閲覧（読み取り専用）するためのマージカレンダーアプリケーション。

## 2. 認証・認可要件
- **Googleアカウントログイン:** ログイン方法はGoogleのOAuth認証のみ。
- **ホワイトリスト方式のアクセス制御:** 事前にシステムに許可された特定の2名のGoogleアカウントでのみログイン可能とし、強固なアクセス制御を行う。

## 3. カレンダー設定・連携要件
### 3.1. Googleカレンダー連携設定
- **同期のトグルスイッチ:** ログインしたユーザーは、ご自身のGoogleカレンダーの予定をマージ画面に「同期して表示するか否か」を、設定画面などから任意に切り替え（ON/OFF）できる。
- **API連携:** 同期がONの場合、Google Calendar APIを利用して当該アカウントの予定データを取得する。

### 3.2. フリカレ連携設定
- **共有URLの入力機能:** ユーザーがフリカレの「共有URL」をシステムに登録・変更できる入力フォームを設ける。
- **スクレイピング取得:** 登録された共有URLへアクセスし、HTML構造から予定データを抽出（スクレイピング）してシステムに取り込む。

## 4. カレンダー表示要件（読み取り専用）
- **マージ表示:** 同期設定がONになっているGoogleカレンダーと、設定されたフリカレの予定を、一つのカレンダービューに統合して表示する。
- **識別表示:** 予定の出所（Googleかフリカレか）が一目で判別できるよう、視覚的な区別（色分けやアイコン等）を行う。
- ※予定の追加・編集・削除機能は実装しない。

## 5. 同期要件
- **自動同期（バッチ処理）:** 毎日 日本時間（JST）の深夜0:00に定期実行ジョブを起動し、設定が有効なGoogleカレンダーのAPI呼び出しとフリカレのスクレイピングを行い、データを最新状態に更新する。
- **手動同期:** 画面上の「同期」ボタンを押下することで、ユーザーの任意のタイミングで最新情報を取得・画面へ反映させることができる。

## 6. 非機能要件・技術的な留意事項
- **スクレイピングの保守性:** フリカレ側のWebサイト仕様変更に備え、スクレイピング失敗時の検知・通知機能や、修正しやすいロジック設計を意識する。
- **レスポンシブ対応:** PC・スマートフォンのブラウザから快適に閲覧・設定変更できるUI設計とする。

## 7. 技術選定およびアーキテクチャ構成案（v1.1）
- **フロントエンド＆API（ホスティング: Vercel）:** Next.js (TypeScript)
  - **スタイリング:** Tailwind CSS (UIコンポーネントライブラリとしてshadcn/uiなどの活用を想定し、モダンかつ迅速な開発を実現)
- **データベース＆認証（ホスティング: Supabase）:** PostgreSQL, Supabase Auth (Google Provider)
- **外部API設定（GCP）:** Google Calendar API
- **定期実行ジョブ＆スクレイピング（GitHub Actions）:** Node.jsスクリプト（Cheerio/Puppeteer等）を深夜0:00(JST)にcron実行しSupabaseのDBを更新。
